1. 왜 코드 사인이 필요한가?

macOS 는 Gatekeeper 라는 보안 검사 시스템을 통해 인터넷에서 다운로드한 모든 실행 파일·인스톨러를 검증합니다. 사인되지 않은 파일을 실행하면:

🛑 "확인되지 않은 개발자" 차단
Apple 은 이 파일이 누가 만들었는지, 변조됐는지 확인할 수 없습니다.
사용자는 시스템 설정 → 개인정보 보호 및 보안 → 그래도 허용 으로 1회 우회 필요.

이게 일반 사용자에게 큰 진입 장벽입니다. 코드 사인 을 하면:

항목사인 안 함사인 + 공증
Gatekeeper❌ 차단✅ 통과
사용자 클릭 수여러 번 (우회 절차)비밀번호 1회
Apple 신원 보증없음인증서로 보증
변조 감지못함해시 검증 자동

3. 사전 준비 — Apple Developer Program

가입

developer.apple.com/programs 에서 가입. 비용은 연 $99 (개인) / $299 (회사).

💡 무료 Apple ID 와 차이
무료 Apple ID 로도 Xcode 로 본인 디바이스에 빌드는 가능하지만, Developer ID 인증서는 유료 멤버십에서만 발급 가능. 외부 배포(앱스토어 밖) 용 사인은 멤버십 필수.

Team ID 확인

가입 후 좌상단 계정명 옆 또는 Membership 페이지에서 확인. 10자리 영숫자 (예: RRTB256N59).

▶ Apple Developer 계정 페이지. 우상단에 본인 이름 + Team ID 표시. "프로그램 리소스" 카드에서 "인증서, ID 및 프로파일" 진입.

현재 보유 인증서 확인

Certificates 페이지에 들어가면 발급된 인증서 목록. 일반 가입자는 Development / Distribution 만 있고 Developer ID 는 없음 — 따로 발급해야 함.

▶ 기본 발급 4개 (Development × 2, Distribution × 2). Developer ID 종류가 없음 — 우상단 [+] 로 별도 발급 필요.
인증서 종류용도외부 배포 가능?
Apple Development개발 빌드 (본인 디바이스)❌
Apple DistributionApp Store 업로드❌ (App Store 만)
Developer ID Application.app / .command / .dylib 사인 → 외부 배포✅
Developer ID Installer.pkg 사인 → 외부 배포✅

4. Developer ID Application 인증서

이게 뭔가요?

Apple 이 발급하는 디지털 신원 증명서. 이 인증서로 사인된 .app/.command/.dylib 는 Apple 의 신원 보증을 받아 Gatekeeper 가 즉시 통과시킵니다.

💡 비유
여권과 비슷합니다 — 본인이 만들 수 없고, 권위 기관(Apple) 이 본인 신원 확인 후 발급. 파일에 사인하는 것 = 여권 도장으로 본인이 만든 거 증명하는 것.

의미·목적

측면설명
신원 보증"이 .app 은 RRTB256N59 팀이 만든 진짜"
변조 감지파일이 한 비트라도 바뀌면 사인 무효화 (해시 검증)
Gatekeeper 통과"확인되지 않은 개발자" 차단 사라짐
공증 전제사인 안 된 파일은 Apple 공증 서버가 거부

왜 필요한가? — 우리 프로젝트 사례

KyoboLibrary 의 워커는 사용자 Mac 에서 백그라운드로 실행되어야 합니다. 인스톨러를 사인 안 하면 사용자가 다운로드할 때마다 차단되어 매번 시스템 설정에서 우회해야 합니다. 사인하면 더블클릭 → 비밀번호 1회 → 자동 설치 끝.

발급 절차

1 Apple Developer Portal → Certificates → 우상단 [+]
2 Software 섹션에서 Developer ID Application 선택 → Continue
3 Profile Type: G2 Sub-CA (Xcode 11.4.1 or later) → Continue
4 CSR 파일 업로드 (다음 섹션 5 참조)
5 Download → .cer 파일 받음 → 더블클릭 → 키체인 등록
▶ "Create a New Certificate" 화면. Profile Type 선택 후 [Choose File] 로 CSR 업로드.
▶ 발급 완료. Certificate Type 이 "Developer ID Application", 만료일 2031/06/02. [Download] 클릭 → .cer 파일 다운로드.

검증 명령어

# 키체인 등록 확인 — codesigning policy
security find-identity -v -p codesigning | grep "Developer ID Application"

# 결과 예:
# 13) B46F5D6A77504BED984712CBB629778879F6CAE4 "Developer ID Application: deok soo yun (RRTB256N59)"

저장 위치 + 어떻게 사용

인증서 (.cer 파일)OneDrive 안 보관 폴더 (백업·재사용)
실제 사용키체인 안 — ~/Library/Keychains/login.keychain-db
비공개키키체인 안 (Mac 본체에만 존재, export 금지)
# 사용 예 — .app 사인
codesign --sign "Developer ID Application: deok soo yun (RRTB256N59)" \
         --options runtime --timestamp YourApp.app

5. 중간 단계 — CSR 만들기 (Keychain Access)

CSR (Certificate Signing Request) 은 "이런 인증서 만들어 주세요" 요청서. 공개키 + 신원 정보(이메일·이름) 가 들어있고, Apple 이 이걸 받아서 검토 후 인증서 발급. 비공개키는 Mac 의 키체인 안에 남아 절대 외부로 나가지 않음.

🔑 비공개키 + 공개키 짝
Keychain 에서 CSR 만들 때 RSA 2048 키쌍이 자동 생성:
  • 비공개키 (Private Key) — Mac 키체인 안. 절대 export X. 사인할 때 필수.
  • 공개키 (Public Key) — CSR 안에 들어가 Apple 에 전송. 인증서에 들어가 배포됨.

준비 조건

  • macOS Mac (Apple Silicon / Intel 무관)
  • Keychain Access.app — macOS Tahoe (26) 이후엔 /System/Library/CoreServices/Applications/Keychain Access.app
  • 본인 Apple ID 이메일

생성 절차

1 Keychain Access 실행 (⌘+Space → "Keychain Access" 또는 한글 "키체인 접근")
2 상단 메뉴 키체인 접근 → 인증서 지원 → 인증 기관에서 인증서 요청...
3 정보 입력 (다음 표 참조)
4 [계속] → 저장 위치 선택 → 파일 저장
5 키 쌍 정보 → 2048비트 / RSA → [계속] → [완료]
▶ 키체인 접근 메뉴 → 인증서 지원 → "인증 기관에서 인증서 요청..." 클릭.

입력값

필드값비고
사용자 이메일 주소본인 Apple ID 이메일예: ckidsm@nate.com
일반 이름본인 이름인증서 CN 으로 사용
CA 이메일 주소(비워두기)Apple 이 무시
요청 처리⚪ 디스크에 저장됨"이메일로 보냄" 아님!
키 쌍 정보 지정✅ 체크RSA 2048 사용을 위해
▶ 인증서 정보 입력 창. "디스크에 저장됨" + "본인이 키 페어 정보 지정" 체크 후 [계속].

CSR 파일 검증

openssl req -in "CSR_xxx.certSigningRequest" -noout -subject -text

# 정상 결과 예:
# subject=emailAddress=ckidsm@nate.com, CN=..., C=KR
# Public Key Algorithm: rsaEncryption
# Public-Key: (2048 bit)
# Signature Algorithm: sha256WithRSAEncryption
⚠ 같은 CSR 재사용은 한 번만
Application 인증서 발급에 한 번 사용된 CSR 은 Installer 발급 시 거부될 수 있습니다. 각 인증서마다 새 CSR 생성을 권장.
에러 메시지: An attribute specified more than once

6. Developer ID Installer 인증서

이게 뭔가요?

.pkg 파일 (인스톨러 패키지) 사인 전용 인증서. Application 인증서가 실행 파일 사인용이라면, 이건 인스톨러 사인용. 둘은 별개.

왜 따로 있나?

도구대상 파일사용 인증서
codesign.app, .command, .dylibDeveloper ID Application
productbuild --sign.pkgDeveloper ID Installer

.pkg 는 인스톨러 메타데이터 + payload + 스크립트 모음이라 사인 방식이 다릅니다. Apple 이 두 종류로 분리해서 발급.

발급 절차 — Application 과 거의 같음

1 Apple Developer Portal → Certificates → [+]
2 Software 섹션 → Developer ID Installer 선택 → Continue
3 G2 Sub-CA → Continue
4 새 CSR 업로드 (Application 에 쓴 CSR 재사용 시 에러)
5 Download → 더블클릭 → 키체인
▶ "An attribute specified more than once" — 같은 CSR 을 두 번 사용했을 때 Apple 이 거부. 새 CSR 만들어서 다시.
▶ Installer 인증서 발급 완료. Certificate Type 이 "Developer ID Installer".

검증 명령어 (주의 — policy 가 다름)

# Application 은 codesigning policy, Installer 는 basic policy
security find-identity -v -p basic | grep "Developer ID Installer"

# 또는 모든 policy 한꺼번에 (가장 간단)
security find-identity -v | grep "Developer ID"

# 결과 예:
# 13) B46F5D6A77504BED... "Developer ID Application: deok soo yun (RRTB256N59)"
# 14) E2F5E229E3702DE2... "Developer ID Installer: deok soo yun (RRTB256N59)"
💡 왜 두 policy 가 다른가
macOS 의 키체인은 인증서의 "사용 목적" 에 따라 policy 분류합니다. Application 은 코드 사인 용도(codesigning), Installer 는 일반 사인 용도(basic). 그래서 -p codesigning 만으로는 Installer 가 안 보입니다.

사용 예시

# .pkg 사인 (productbuild 또는 pkgbuild)
productbuild --distribution dist.xml \
             --sign "Developer ID Installer: deok soo yun (RRTB256N59)" \
             output.pkg

7. notarytool credential — 공증 자격증명

공증(Notarization) 이란?

사인만으로는 부족합니다 — Apple 은 추가로 "이 파일에 멀웨어가 없다" 를 직접 검사합니다. 이 검사 과정이 공증(Notarization). 사인된 파일을 Apple 서버에 업로드 → 악성코드 스캔 → 통과 시 "Notarized" 도장.

credential 이 왜 필요?

공증 요청 시 Apple 서버에 인증해야 합니다. 인증 방법:

인증 방식설명사용 빈도
App-specific password본인 Apple ID + 별도 비밀번호 1쌍가장 일반적
Developer ID API KeyApp Store Connect API 키 발급CI/CD 자동화
로그인 세션매번 Apple ID 비밀번호 + 2FA 입력비효율적 — 거의 안 씀
💡 왜 본인 Apple ID 비밀번호 직접 안 씀
Apple 은 보안상 "앱별 비밀번호" 를 권장. 본인 비밀번호 노출 위험 X, 필요시 개별 폐기 가능.

App-specific password 생성

1 appleid.apple.com 접속 → 로그인
2 좌측 메뉴 로그인 및 보안
3 우측 화면 → 앱 암호 섹션 → [+] 또는 [생성]
4 이름 입력 (예: notarytool) → 생성 → 본인 Apple ID 비밀번호 인증
5 ⚠ 16자리 비밀번호 표시 (단 한 번!) — 창 닫기 전에 다음 단계까지 진행
▶ 앱 암호 화면. 이미 생성된 notarytool 항목이 보임. ⊖ 로 삭제, [+] 로 새로 생성.

keychain 에 credential 저장

xcrun notarytool store-credentials "KYOBO_NOTARY" \
    --apple-id ckidsm@nate.com \
    --team-id RRTB256N59

# 실행 후 프롬프트:
# App-specific password for ckidsm@nate.com: xxxx-xxxx-xxxx-xxxx
# Validating your credentials...
# Success. Credentials validated.
# Credentials saved to Keychain.

이후 모든 공증 요청에서 이 profile 이름만 참조:

xcrun notarytool submit my.pkg --keychain-profile "KYOBO_NOTARY" --wait
✓ 한 번만 등록하면 끝
이후 모든 빌드 스크립트에서 --keychain-profile "KYOBO_NOTARY" 만 쓰면 됨. 비밀번호 다시 입력할 일 없음.

8. .pkg 인스톨러란?

정의

macOS 표준 인스톨러 패키지. 사용자가 더블클릭하면 Installer.app 이 자동 실행되어 표준 UI 로 설치 진행.

어디에 사용?

  • 앱 설치 (App Store 안 거치는 외부 배포)
  • 드라이버·plugin 설치
  • 시스템 설정 변경 (LaunchDaemons 등록 등)
  • 여러 파일을 정해진 위치에 한꺼번에 복사
  • 설치 전/후 스크립트 실행

핵심 명령어

① pkgbuild (component pkg 만들기)

pkgbuild \
    --root "$PAYLOAD_ROOT" \
    --identifier "com.kyobolibrary.worker" \
    --version "1.0.0" \
    --scripts "$SCRIPTS_DIR" \
    --install-location "/" \

옵션의미
--root설치할 파일들이 들어있는 디렉토리. 이 안의 파일 구조 = 사용자 시스템에 그대로 복사됨

postinstall 스크립트 (핵심 트릭)

.pkg postinstall 은 root 권한 으로 실행됩니다. 하지만 사용자 컨텍스트의 launchd 에 등록하려면 사용자 권한 이 필요. 그래서 sudo -u 로 컨텍스트 전환: 

#!/bin/bash
# root 컨텍스트에서 콘솔 사용자 찾기
CONSOLE_USER=$(stat -f "%Su" /dev/console)
USER_HOME=$(dscl . -read /Users/"$CONSOLE_USER" NFSHomeDirectory | awk '{print $2}')

# 사용자 권한으로 install-worker-macos.sh 실행
sudo -u "$CONSOLE_USER" -H bash -c "
    export HOME='$USER_HOME'
    export PATH='/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin'
    '$USER_HOME/.../install-worker-macos.sh'
"

11. 자주 만나는 에러 + 트러블슈팅

① "An attribute specified more than once"

증상
Apple Developer 포털에서 CSR 업로드 시 빨간 알림.

원인: 같은 CSR 을 두 번째 인증서 발급에 재사용.

해결: Keychain Access 에서 새 CSR 생성. 기존 CSR 은 보관(감사 추적용).

② security find-identity 에 Installer 가 안 보임

원인: Installer 는 basic policy 라 -p codesigning 으로 안 검색됨.

# 둘 다 보려면 -p 빼고
security find-identity -v | grep "Developer ID"

③ "notarytool: command not found"

원인: Xcode CLT 미설치. 또는 PATH 누락.

# 설치
xcode-select --install

# 또는 xcrun 으로 호출
xcrun notarytool ...

④ launchd 로 실행되는 Python 의 PATH 누락

원인: launchd plist 의 EnvironmentVariables.PATH 가 시스템 기본과 다름.

# /usr/sbin 누락 시 screencapture 못 찾음 → 무한 실패
<key>PATH</key>
<string>/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin</string>

⑤ 공증 status: Invalid

원인: 사인 안 된 바이너리 포함, hardened runtime 미설정, entitlements 누락 등.

# 상세 로그 확인
xcrun notarytool log <submission-id> \
    --keychain-profile "KYOBO_NOTARY"

⑥ "Developer ID 인증서가 키체인에 없음"

원인: .cer 파일을 더블클릭만 하면 안 됨. 실제로는 비공개키와 짝지어진 인증서 만 사인에 쓸 수 있음.

비공개키는 CSR 만든 Mac 의 키체인에 있어야 합니다 — 다른 Mac 에서 사인하려면 p12 export 필요.
