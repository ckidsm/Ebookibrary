주요 주제
프락시 요청 헤더응답 헤더협상 헤더응답 보안 헤더엔터티 헤더
주요 용어
Max-ForwardsProxy-AuthorizationAgeAccept-RangesWWW-Authenticate
강의 요약

프락시가 흔해지면서 프락시 기능을 돕기 위한 헤더들이 정의되었으며, Max-Forwards, Proxy-Authorization, Proxy-Connection 등이 있다.
응답 헤더는 클라이언트에게 부가 정보를 제공하며, Age, Server, Warning 등을 통해 응답자의 능력과 응답에 대한 설명을 제공한다.
협상 헤더는 서버와 클라이언트가 여러 표현 중 어떤 것을 택할지 협상할 수 있도록 지원한다.
응답 보안 헤더는 HTTP 인증요구/응답 체계에서 사용되며, WWW-Authenticate, Set-Cookie 등이 포함된다.
엔터티 헤더는 요청과 응답 양쪽 모두에 나타날 수 있으며, 엔터티와 그 내용물에 대한 광범위한 정보를 제공한다.

핵심 내용
  • Max-Forwards: 요청이 원 서버로 향하는 과정에서 다른 프락시나 게이트웨이로 전달될 수 있는 최대 횟수를 지정하며, TRACE 메서드와 함께 사용
  • Proxy-Authorization: Authorization과 같으나 프락시에서 인증을 할 때 사용
  • Age 헤더: 응답이 얼마나 오래되었는지를 나타내며, 응답이 중개자나 프락시를 통해 왔음을 암시
  • Accept-Ranges: 서버가 자원에 대해 받아들일 수 있는 범위의 형태를 명시
  • WWW-Authenticate: 서버에서 클라이언트로 보낸 인증요구의 목록을 포함하는 응답 보안 헤더

주요 주제
HTTP 엔터티 헤더콘텐츠 헤더와 캐싱 헤더
주요 용어
엔터티 정보 헤더콘텐츠 헤더엔터티 캐싱 헤더Content-TypeETag
강의 요약

HTTP 엔터티 헤더는 리소스의 특정 버전에 대한 정보를 제공한다.
엔터티 정보 헤더는 Allow, Location, ETag, Expires, Last-Modified 등을 포함하며 엔터티의 위치와 유효성 정보를 제공한다.
콘텐츠 헤더는 Content-Type, Content-Length, Content-Encoding 등으로 본문의 종류, 크기, 인코딩 정보를 기술한다.
엔터티 캐싱 헤더는 캐시된 사본의 유효성과 만료 시점에 대한 정보를 제공한다.

핵심 내용
  • 엔터티 정보 헤더: Allow(수행 가능한 요청 메서드), Location(실제 위치), ETag(엔터티 태그), Expires(만료 일시), Last-Modified(최근 변경 일시)
  • 콘텐츠 헤더: Content-Type(객체 종류), Content-Length(본문 길이), Content-Encoding(인코딩), Content-Language(자연어), Content-Range(범위)
  • 엔터티 캐싱 헤더: 캐시된 사본의 유효성 정보와 만료 시점 추정을 위한 단서 제공
  • 브라우저는 Content-Type 헤더를 보고 객체를 어떻게 표시할지 결정

HTTP 커넥션과 TCPHTTP 통신 과정
주요 용어
TCP 커넥션HTTP 커넥션패킷 교환 네트워크신뢰할 수 있는 데이터 전송
강의 요약

· 4장 HTTP 커넥션 관리의 개요와 학습 목표를 소개한다.
HTTP는 TCP/IP를 통해 통신하며, TCP 커넥션을 통해 메시지를 안전하게 전달한다.
· 4.1 TCP 커넥션에서는 HTTP 통신이 TCP를 기반으로 이루어지는 과정을 설명한다.
브라우저가 URL에서 호스트명과 포트를 추출하고, IP 주소를 찾아 TCP 커넥션을 맺은 후 HTTP 메시지를 주고받는 7단계 과정을 제시한다.
· 4.1.1에서는 TCP가 신뢰할 수 있는 데이터 전송 통로임을 강조한다.

핵심 내용
  • 4장에서 다루는 내용: HTTP의 TCP 커넥션 사용 방식, TCP 커넥션의 지연·병목·막힘, 병렬 커넥션·keep-alive·파이프라인을 활용한 최적화, 커넥션 관리 규칙
  • HTTP 통신의 7단계: (1) 호스트명 추출 → (2) IP 주소 조회 → (3) 포트 번호 획득 → (4) TCP 커넥션 생성 → (5) HTTP 요청 전송 → (6) HTTP 응답 수신 → (7) 커넥션 종료
  • TCP 커넥션의 특징: 메시지가 손실·손상되거나 순서가 바뀌지 않고 안전하게 전달되는 신뢰할 수 있는 데이터 전송 통로
  • HTTP 커넥션은 몇몇 사용 규칙을 제외하고는 TCP 커넥션에 불과하며, HTTP 애플리케이션 개발을 위해서는 TCP의 기초를 이해해야 함

주요 주제
TCP 연결과 데이터 전송IP 패킷과 세그먼트 구조TCP/IP 프로토콜 스택
주요 용어
TCPIP 패킷세그먼트프로토콜 스택포트 번호
강의 요약

·4.1.2 TCP는 데이터를 세그먼트 단위로 나누어 IP 패킷에 담아 전송한다.
HTTP는 IP, TCP, HTTP로 구성된 프로토콜 스택의 최상위 계층이며, HTTPS는 TCP와 HTTP 사이에 TLS/SSL 암호화 계층을 둔다.
각 IP 패킷은 IP 패킷 헤더(20바이트), TCP 세그먼트 헤더(20바이트), TCP 데이터 조각으로 구성된다.
·4.1.3 컴퓨터는 여러 개의 TCP 커넥션을 포트 번호를 통해 유지한다.

핵심 내용
  • TCP 스트림 전송 구조: HTTP 메시지 → TCP 세그먼트로 분할 → IP 패킷에 담아 전송
  • 프로토콜 스택 계층: HTTP(애플리케이션) - TCP(전송) - IP(네트워크) - 데이터링크, HTTPS는 HTTP와 TCP 사이에 TLS/SSL 보안 계층 추가
  • IP 패킷 구성요소: IP 헤더(발신지/목적지 IP 주소, 크기, 플래그), TCP 헤더(포트 번호, 제어 플래그, 순서/무결성 검사 값), TCP 데이터
  • TCP 포트 번호: 하나의 컴퓨터에서 여러 TCP 커넥션을 구분하고 유지하는 식별자

주요 주제
TCP 커넥션 식별소켓 프로그래밍
주요 용어
TCP 커넥션포트 번호소켓 API발신지/수신지 주소TCP 종단
강의 요약

·4.1.4 TCP 소켓 프로그래밍: TCP 커넥션은 발신지 IP 주소, 발신지 포트, 수신지 IP 주소, 수신지 포트 네 가지 값으로 유일하게 식별된다.
운영체제는 TCP 커넥션 생성을 위한 소켓 API를 제공하며, 이는 TCP와 IP의 세부사항을 숨긴다.
소켓 API는 유닉스용으로 먼저 개발되었지만 현재는 대부분의 운영체제와 프로그램 언어에서 사용 가능하다.
주요 소켓 함수로는 socket, bind, connect, listen, accept, read, write, close 등이 있다.

핵심 내용
  • TCP 커넥션 식별: 발신지 IP 주소, 발신지 포트, 수신지 IP 주소, 수신지 포트 네 가지 값으로 유일한 커넥션 생성
  • 소켓 API 함수: socket(소켓 생성), bind(로컬 포트 할당), connect(커넥션 생성), listen(연결 대기), accept(연결 수락), read/write(데이터 송수신), close(커넥션 종료)
  • 소켓 API의 역할: TCP/IP의 세부사항, 핸드셰이킹, 데이터 스트림 분할 및 재조립 등을 외부로부터 숨김
  • 웹브라우저는 소켓 API를 이용하여 URL에서 IP 주소와 포트 번호를 알아내고 서버에 TCP 커넥션을 생성

주요 주제
TCP 소켓 인터페이스HTTP 트랜잭션 지연TCP 성능 고려사항
주요 용어
소켓커넥션HTTP 트랜잭션TCP 지연DNS 이름 분석
강의 요약

클라이언트와 서버는 TCP 소켓 인터페이스를 통해 상호작용하며, 서버는 소켓 생성·바인딩·리스닝 과정을 거치고 클라이언트는 연결 후 HTTP 요청을 전송한다.
· 4.2 HTTP 트랜잭션의 성능은 하위 계층인 TCP 성능에 영향을 받으며, 대부분의 HTTP 지연은 TCP 네트워크 지연 때문에 발생한다.
· 4.2.1 HTTP 트랜잭션을 지연시키는 주요 원인은 DNS 이름 분석(수십 초), TCP 커넥션 설정(1~2초), 요청/응답 전송 시간이다.
클라이언트나 서버가 대량 데이터를 처리하지 않는 한, 실제 트랜잭션 처리 시간은 TCP 커넥션 설정 및 전송 시간에 비해 상당히 짧다.

핵심 내용
  • TCP 소켓 통신 과정: 서버는 소켓 생성(S1) → 포트 바인딩(S2) → 커넥션 허가(S3) → 대기(S4), 클라이언트는 주소 획득(C1) → 소켓 생성(C2) → 연결(C3) 순서로 진행
  • DNS 이름 분석 지연: URI의 호스트명을 IP 주소로 변환하는 데 수십 초 소요 가능 (최근 방문 기록 없을 시)
  • TCP 커넥션 설정 지연: 새로운 TCP 커넥션마다 1~2초 소요되며, 수백 개의 HTTP 트랜잭션 시 누적 시간 증가
  • HTTP 지연의 주요 원인: 트랜잭션 처리 시간보다 TCP 네트워크 지연(커넥션 설정, 전송)이 대부분의 지연 발생

주요 주제
TCP 커넥션 성능HTTP 트랜잭션 지연TCP 핸드셰이크
주요 용어
TCP 핸드셰이크SYNACK네이글 알고리즘느린 시작
강의 요약

· 4.2.2 HTTP 성능에 영향을 주는 TCP 관련 지연 요소로는 핸드셰이크 설정, 느린 시작, 네이글 알고리즘, 확인응답 지연, TIME_WAIT 지연 등이 있다.
· 4.2.3 새로운 TCP 커넥션을 열 때마다 SYN, SYN+ACK, ACK 패킷을 순서대로 교환하는 핸드셰이크 과정이 필요하다.
작은 크기의 HTTP 트랜잭션은 50% 이상의 시간을 TCP 구성에 소비하여 성능 저하가 발생한다.
HTTP는 이러한 지연을 줄이기 위해 기존 커넥션을 재활용하는 방법을 사용한다.

핵심 내용
  • TCP 성능 관련 주요 요소: 핸드셰이크 설정, 느린 시작(slow-start), 네이글 알고리즘, 확인응답 지연, TIME_WAIT 지연과 포트 고갈
  • TCP 핸드셰이크 3단계: ① 클라이언트가 SYN 플래그를 가진 패킷 전송 ② 서버가 SYN+ACK 플래그 패킷으로 응답 ③ 클라이언트가 ACK 확인응답 전송
  • 작은 HTTP 트랜잭션의 경우 50% 이상의 시간을 TCP 구성에 소비하여 핸드셰이크 지연이 성능에 큰 영향을 미침
  • ACK 패킷은 HTTP 요청 메시지 전체를 전달할 수 있을 만큼 크며, 많은 서버 응답도 하나의 IP 패킷에 담길 수 있음

주요 주제
네이글 알고리즘과 HTTP 성능TIME_WAIT 포트 고갈HTTP 커넥션 관리
주요 용어
네이글 알고리즘TCP_NODELAYTIME_WAIT포트 고갈2MSL
강의 요약

네이글 알고리즘은 작은 HTTP 메시지를 지연시키고 확인응답 지연과 함께 사용 시 성능 문제를 발생시키므로, HTTP 애플리케이션은 TCP_NODELAY 파라미터로 이를 비활성화하기도 한다.
TCP 커넥션 종료 시 TIME_WAIT 상태로 인해 같은 주소와 포트의 재사용이 2분(2MSL) 동안 제한되며, 성능 테스트 환경에서는 포트 고갈 문제가 발생할 수 있다.
제한된 발신지 포트(약 60,000개)와 TIME_WAIT 시간(120초)으로 인해 초당 약 500개의 커넥션으로 제한되며, 이는 성능 측정 시 심각한 병목이 될 수 있다.
포트 고갈 문제 해결을 위해 부하 생성 장비를 늘리거나 여러 가상 IP 주소를 사용할 수 있다.

핵심 내용
  • 네이글 알고리즘 문제: 작은 HTTP 메시지 지연, 확인응답 지연(100~200ms)과 함께 사용 시 성능 저하
  • TCP_NODELAY 파라미터: 네이글 알고리즘 비활성화로 성능 개선, 단 큰 데이터 덩어리 생성 필요
  • TIME_WAIT 포트 고갈: 2MSL(약 2분) 동안 같은 주소/포트 재사용 불가, 성능 테스트 시 초당 500개 커넥션 제한
  • 포트 고갈 해결책: 부하 생성 장비 추가, 여러 가상 IP 주소 사용으로 가능한 연결 조합 확대

주요 주제
Connection 헤더순차적 트랜잭션 처리의 지연
주요 용어
Connection 헤더hop-by-hop커넥션 토큰순차적 트랜잭션TCP 성능
강의 요약

· 4.3.1 Connection 헤더는 현재 커넥션에만 적용될 옵션을 지정하며, HTTP 헤더 필드명/임시 토큰/close 값 세 가지 종류의 토큰을 전달할 수 있다.
Connection 헤더에 명시된 헤더들은 hop-by-hop 헤더로, 다음 커넥션으로 전달되지 않고 삭제되어야 한다.
· 4.3.2 순차적 트랜잭션 처리는 각 트랜잭션마다 새로운 커넥션을 맺어 지연과 느린 시작 문제를 발생시킨다.
여러 이미지가 있는 웹페이지를 순차적으로 로드하면 물리적 지연과 심리적 지연이 모두 발생한다.

핵심 내용
  • Connection 헤더의 세 가지 토큰: HTTP 헤더 필드명(현재 커넥션만 해당), 임시 토큰(비표준 옵션), close(작업 완료 후 종료)
  • hop-by-hop 헤더: Connection 헤더에 명시된 헤더는 다음 hop으로 전달 시 삭제되며, Proxy-Authenticate, Transfer-Encoding 등도 hop-by-hop 헤더에 해당
  • 순차적 처리의 문제: 네 개의 트랜잭션이 순차적으로 처리되면 각각 커넥션 맺기 지연과 느린 시작 지연 발생
  • 사용자는 이미지를 하나씩 순차적으로 내려받는 것보다 여러 개가 동시에 로드되는 방식을 더 선호

  • 지속 커넥션의 장점: 커넥션 준비 시간 절약, TCP 느린 시작 회피, 커넥션 수 감소

·Keep-Alive 헤더는 max(트랜잭션 수)와 timeout(유지 시간) 파라미터를 통해 커넥션 유지 정보를 전달한다.

핵심 내용
  • Keep-Alive 헤더 파라미터: max는 처리할 트랜잭션 수, timeout은 유지 시간(초)을 의미
  • 커넥션 유지 조건: 엔터티 본문의 정확한 Content-Length 또는 멀티파트 미디어 타입, 청크 전송 인코딩 필요
  • 프락시 규칙: Connection 헤더에 명시된 모든 헤더 필드와 Connection 헤더 자체를 전달 전에 제거해야 함

  • 클라이언트 준비사항: 응답 전체를 받기 전에 커넥션이 끊어진 경우 요청을 다시 보낼 수 있어야 함

  • 영리한 프락시: Proxy-Connection 헤더를 인식하고 Connection: Keep-Alive 헤더로 변환하여 서버에 전송

HTTP 요청 재시도와 멱등성TCP 커넥션 종료 방식우아한 커넥션 끊기
주요 용어
멱등성(idempotent)전체 끊기절반 끊기close()shutdown()connection reset by peer
강의 요약

멱등성(idempotent)은 한 번 또는 여러 번 실행해도 같은 결과를 반환하는 특성으로, GET, HEAD, PUT, DELETE, TRACE, OPTIONS 메서드가 이에 해당한다.
POST와 같이 멱등이 아닌 요청은 파이프라인을 통해 요청하면 안 되며, 재시도 시 이전 응답을 받을 때까지 기다려야 한다.
TCP 커넥션 종료 방식에는 close()를 사용한 '전체 끊기'와 shutdown()을 사용한 '절반 끊기'가 있다.
입력 채널을 먼저 끊으면 클라이언트가 데이터를 전송할 때 'connection reset by peer' 에러가 발생하여 버퍼의 데이터가 삭제될 수 있다.
파이프라인 지속 커넥션에서는 예상치 못한 쓰기 에러를 방지하기 위해 절반 끊기를 사용해야 한다.

핵심 내용
  • 멱등성 메서드: GET, HEAD, PUT, DELETE, TRACE, OPTIONS는 여러 번 실행해도 같은 결과를 반환
  • POST 요청 주의: 멱등이 아닌 요청은 파이프라인 사용 금지, 재시도 시 이전 응답 대기 필요
  • 전체 끊기: close() 호출로 입력/출력 채널 모두 종료
  • 절반 끊기: shutdown() 호출로 입력 또는 출력 채널 중 하나만 종료
  • 입력 채널 종료 위험: 클라이언트가 끊긴 입력 채널에 데이터 전송 시 'connection reset by peer' 에러 발생, 버퍼 데이터 삭제

TCP 확인응답 지연TCP 느린 시작네이글 알고리즘
주요 용어

강의 요약

·4.2.4 TCP는 데이터 무결성 보장을 위해 확인응답 체계를 사용하며, 확인응답 지연 알고리즘은 0.1~0.2초 동안 확인응답을 버퍼에 저장해 송출 데이터 패킷에 편승시킨다.
HTTP의 요청-응답 구조는 확인응답이 편승할 기회를 감소시켜 지연을 자주 발생시킨다.
·4.2.5 TCP 느린 시작은 커넥션 초기에 전송 패킷 수를 제한하고, 성공적 전달 시점마다 전송 권한을 늘려가며 혼잡을 방지한다.
·4.2.6 네이글 알고리즘은 작은 데이터를 버퍼에 모아 최대 크기의 세그먼트로 합쳐 전송함으로써 네트워크 효율을 높인다.

핵심 내용
  • 확인응답 지연(Delayed ACK): 확인응답을 0.1~0.2초 버퍼에 저장하여 송출 데이터 패킷에 piggyback 시도
  • TCP 느린 시작(Slow Start): 초기 1개 패킷 전송 후 확인응답마다 전송 가능 패킷 수를 2배씩 증가(혼잡 윈도 열기)
  • 네이글 알고리즘: 세그먼트가 최대 크기(1,500바이트)가 되거나 모든 패킷이 확인응답 받을 때까지 전송 지연
  • HTTP 요청-응답 구조는 확인응답 편승 기회를 감소시켜 지연 발생 빈도 증가
  • 튜닝된 커넥션이 새 커넥션보다 빠르므로 HTTP는 기존 커넥션 재사용(지속 커넥션) 기능 제공

Keep-Alive지속 커넥션Connection 헤더timeout 파라미터핸드셰이크
강의 요약

· 4.5.2 HTTP/1.0+는 실험적인 keep-alive 커넥션을 통해 지속 커넥션을 지원하며, 연속 커넥션 대비 커넥션 생성/종료 오버헤드를 제거하여 성능을 향상시킨다.
· 4.5.3 Keep-Alive는 HTTP/1.1 명세에서 빠졌지만 여전히 널리 사용되며, 클라이언트가 Connection: Keep-Alive 헤더를 보내면 서버도 같은 헤더로 응답하여 커넥션을 유지한다.
· 4.5.4 Keep-Alive 헤더는 요청일 뿐 강제가 아니며, timeout 파라미터 등의 옵션으로 동작을 제어할 수 있다.

핵심 내용
  • Keep-Alive 성능 이점: 네 개의 HTTP 트랜잭션 처리 시 연속 커넥션 대비 하나의 지속 커넥션 사용으로 커넥션 생성/종료 작업 제거
  • Keep-Alive 핸드셰이크: 클라이언트가 Connection: Keep-Alive 헤더를 요청에 포함하고, 서버가 응답에 같은 헤더를 포함하면 커넥션 유지
  • Keep-Alive 옵션: timeout 파라미터로 커넥션 유지 시간 지정 가능하나 동작 보장은 없음
  • 서버 응답에 Connection: Keep-Alive 헤더가 없으면 클라이언트는 응답 후 커넥션이 끊길 것으로 판단
  • 클라이언트나 서버는 keep-alive 요청을 받아도 무조건 따를 필요 없으며, 언제든지 커넥션을 끊거나 트랜잭션 수를 제한 가능

Keep-Alive와 프락시 상호작용 문제Connection 헤더 전달 규칙Proxy-Connection 헤더
주요 용어
Keep-Alive멍청한 프락시(dumb proxy)홉별 헤더(hop-by-hop header)Connection 헤더Proxy-Connection
강의 요약

· HTTP/1.0 Keep-Alive를 이해하지 못하는 멍청한 프락시는 Connection: Keep-Alive 헤더를 서버로 그대로 전달하여 문제를 발생시킨다.
Connection 헤더는 홉별(hop-by-hop) 헤더로 한 개의 전송 링크에만 적용되며 다음 서버로 전달되어서는 안 된다.
프락시가 Connection 헤더를 전달하면 클라이언트-프락시-서버 간 커넥션 상태 불일치가 발생하고, 클라이언트의 두 번째 요청이 무시되어 행에 걸린다.
프락시는 Connection 헤더와 그 헤더에 명시된 헤더들(Keep-Alive 등)을 절대 전달하면 안 되며, Proxy-Authenticate, Transfer-Encoding 등의 홉별 헤더도 전달하거나 캐시하면 안 된다.
넷스케이프는 Proxy-Connection 헤더를 차선책으로 제시했으나 특정 상황에서만 동작한다.

핵심 내용
  • 멍청한 프락시 문제: Keep-Alive를 이해하지 못하는 프락시가 Connection 헤더를 그대로 전달하여 클라이언트-서버 간 커넥션 상태 불일치 발생
  • 홉별 헤더 규칙: Connection 헤더는 한 개의 전송 링크에만 적용되며 다음 서버로 전달 금지
  • 프락시 전달 금지 헤더: Connection에 명시된 헤더, Proxy-Authenticate, Proxy-Connection, Transfer-Encoding, Upgrade 등은 전달하거나 캐시 금지
  • Proxy-Connection 헤더: 넷스케이프가 제시한 차선책으로, 현대 브라우저와 많은 프락시에서 지원하지만 모든 상황에서 동작하지는 않음

커넥션 끊기Content-Length파이프라인 커넥션
주요 용어
커넥션 끊기Content-LengthTruncation파이프라인재시도
강의 요약

· 4.7 커넥션 끊기에 대한 미스터리: HTTP 커넥션 관리에는 명확한 기준이 없으며, 클라이언트/서버/프락시는 언제든지 TCP 커넥션을 끊을 수 있다.
· 4.7.1 '마음대로' 커넥션 끊기: 커넥션은 메시지 전송 후 끊기지만 에러 상황에서는 헤더 중간에도 끊길 수 있으며, 지속 커넥션은 유휴 상태에서 임의로 끊길 수 있다.
· 4.7.2 Content-Length와 Truncation: 각 HTTP 응답은 정확한 Content-Length 헤더를 가져야 하며, 길이 불일치 시 수신자는 서버에 확인해야 한다.
· 4.7.3 커넥션 끊기의 허용과 재시도: 예상치 못한 커넥션 끊김에 대비해야 하며, 클라이언트는 문제없는 트랜잭션을 재시도할 수 있어야 한다.

핵심 내용
  • 임의 커넥션 끊기: HTTP 애플리케이션은 언제든지 지속 커넥션을 임의로 끊을 수 있으며, 유휴 상태의 커넥션도 끊을 수 있음
  • Content-Length 검증: 커넥션 끊김 후 실제 엔터티 길이와 Content-Length 값이 불일치하면 서버에 확인 필요, 캐시 프락시는 응답을 캐시하면 안 됨
  • 파이프라인 커넥션의 복잡성: 서버가 처리되지 않은 요청을 남겨둔 채 커넥션을 끊을 수 있어 재시도 처리가 어려움
  • 트랜잭션 재시도: 전송 중 커넥션이 끊기면 문제가 없는 경우 커넥션을 다시 맺고 재시도해야 함
